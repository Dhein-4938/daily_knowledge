Graph Neural Networks✓ reviewed

Tech · Machine Learning  ·  ~4 min read

Core Concept

INFO
A Graph Neural Network (GNN) is a neural architecture that operates directly on graph-structured inputs $G = (V, E, \mathbf{X})$, where $V$ is the node set, $E$ the edge set, and $\mathbf{X} \in \mathbb{R}^{|V| \times d}$ the node feature matrix.

The defining mechanism is pairwise message passing: each node $v$ iteratively aggregates representations from its neighborhood $\mathcal{N}(v)$ to update its own embedding. A single layer takes the general form:

$$\mathbf{h}_v^{(k)} = \phi\!\left(\mathbf{h}_v^{(k-1)},\; \bigoplus_{u \in \mathcal{N}(v)} \psi\!\left(\mathbf{h}_v^{(k-1)}, \mathbf{h}_u^{(k-1)}, \mathbf{e}_{uv}\right)\right)$$

where $\psi$ is the message function, $\bigoplus$ is a permutation-invariant aggregator (sum, mean, max), and $\phi$ is the update function. After $K$ layers, the $K$-hop neighborhood of each node has been incorporated.

Two symmetry properties are load-bearing:

  • Permutation equivariance (node-level): reorder nodes → representations reorder identically.
  • Permutation invariance (graph-level): a readout $R(\{\mathbf{h}_v^{(K)}\}_{v \in V})$ must be insensitive to node ordering — typically implemented via a summing or max-pooling readout followed by an MLP.

Why It Matters

Graphs are the natural representation for a vast range of structured data: molecules (atoms = nodes, bonds = edges), social networks, citation graphs, knowledge bases, physical simulations, and combinatorial optimization instances. CNNs and RNNs impose a grid or sequence topology that is artificial for these domains.

TIP
GNNs unify apparently distinct architectures. A CNN layer is a GNN on a grid graph where each pixel connects only to its spatial neighbors. A Transformer layer is a GNN on a complete graph — every token attends to every other — making self-attention a special case of message passing with learned, edge-wise weights. This geometric deep learning perspective cleanly explains why both architectures generalize across scales.

The molecular drug design use case is paradigmatic: predict a molecule's bioactivity (e.g., against E. coli) from its graph structure and atom-level chemical features, where inputs vary in size across compounds. No fixed-size representation is possible without GNNs or ad hoc featurization.

Key Details

  1. Expressive power and the WL test. The expressive power of message-passing GNNs is bounded by the 1-dimensional Weisfeiler-Lehman (1-WL) graph isomorphism test. Two non-isomorphic graphs that 1-WL cannot distinguish will produce identical representations — a hard ceiling for standard architectures.
  1. Over-smoothing. Stacking many message-passing layers causes node representations to converge (all nodes look the same), since each layer effectively blurs features across larger neighborhoods. Practical GNNs are often shallow (2–5 layers) relative to sequence models.
  1. Augmented message passing. Many architectures marketed as "beyond message passing" (e.g., those using higher-order structural features or virtual nodes) can be reframed as message passing on a modified graph. The 2022 position paper term augmented message passing captures this: the graph topology is enriched, not the mechanism itself.
  1. Edge and global features. Real graphs often carry edge attributes (e.g., bond type, distance) and a global graph state. Architectures like MPNN and GN-framework incorporate these explicitly in $\mathbf{e}_{uv}$ and a graph-level latent $\mathbf{u}$.
  1. Scalability. Full neighborhood aggregation is $\mathcal{O}(|E|)$ per layer, but on dense graphs or deep stacks this is prohibitive. Neighbor sampling (GraphSAGE), cluster-based batching (Cluster-GCN), and sparse attention are active engineering mitigations.

Current State

Note: this information may be dated

Position papers continue to argue about whether architectural complexity is necessary or whether expressive augmented message passing suffices for most tasks.

WebAssembly✓ reviewed

Tech · Web Technologies  ·  ~4 min read

Core Concept

INFO
WebAssembly (Wasm) is a portable, binary instruction format for a stack-based virtual machine. It defines a compact binary encoding (.wasm) and a human-readable text format (.wat), along with a formal embedding API — the JavaScript API and Web API — for host-environment integration.

Wasm is not a traditional ISA; it targets a virtual Harvard-architecture machine with linear memory, a value-typed operand stack, and structured control flow (no arbitrary jumps — only blocks, loops, and branches). Programs execute in a sandboxed environment with no implicit access to the host.

The core type system is minimal:

$$\text{valtype} ::= \mathtt{i32} \mid \mathtt{i64} \mid \mathtt{f32} \mid \mathtt{f64}$$

Extended proposals add v128 (SIMD), funcref/externref (reference types), and garbage-collected structs (GC proposal). The module format bundles typed functions, a linear memory, tables, globals, and import/export declarations into a self-describing unit.

Why It Matters

Wasm closes the gap between native performance and safe, portable execution. Key use cases:

  • Browser compute: Video codecs, game engines, physics simulations, and cryptographic libraries run near-native speed without plugins.
  • Edge / serverless: Runtimes like Wasmtime, WASMer, and WAMR execute Wasm outside browsers with <1 ms cold-start times — a decisive advantage over container-based functions.
  • Language universality: C, C++, Rust, Go, Kotlin, and Python all compile to Wasm; the module format is language-agnostic by design.
  • Plugin architectures: Wasm's sandbox makes it an ideal untrusted plugin substrate (see Envoy filters, Extism, and VS Code's WASI-based extension host).
TIP
Wasm's structured control flow is not a limitation — it's what enables one-pass validation in $O(n)$ time and makes JIT compilation significantly simpler than for x86. The spec mandates that any compliant engine must validate a module before execution, making supply-chain exploits structurally harder.

Key Details

1. Linear Memory Model

Wasm memory is a contiguous, byte-addressable ArrayBuffer (initially 0–4 GiB in wasm32, extended to 64-bit in wasm64). Bounds checks are explicit in the bytecode; hardware memory protection via guard pages is a common JIT optimization. Pointer arithmetic is therefore auditable, unlike native binaries.

2. WASI — The System Interface

The WebAssembly System Interface (WASI) provides a capability-based POSIX-like API for file I/O, sockets, clocks, and randomness. Rather than ambient authority, WASI passes pre-opened file descriptors explicitly — a deliberate application of the principle of least privilege at the ABI layer. WASI Preview 2 (Component Model) restructures this around typed interfaces described in WIT (Wasm Interface Type).

3. The Component Model

The Component Model lifts Wasm from a low-level binary format to a composable module system. Components expose and consume typed interfaces (records, variants, streams) using the Canonical ABI, enabling language-agnostic linking without FFI boilerplate.

4. Compilation Tiers

Production engines (V8, SpiderMonkey, Wasmtime) use a two-tier strategy: a fast baseline compiler (Liftoff / Cranelift single-pass) for immediate execution, and an optimizing compiler (TurboFan / Cranelift full) running concurrently. This amortizes compilation latency while converging to near-native throughput.

5. Security Boundary

The sandbox enforces memory isolation, control-flow integrity (structured CFG), and type safety at the bytecode level. Spectre-class attacks remain a concern — engines mitigate via retpoline, serializing memory accesses, and disabling SharedArrayBuffer/Atomics under threat models.

Note: this information may be dated — the Component Model and WASI Preview 2 ecosystem were still maturing as of early 2026; toolchain support (especially for languages beyond Rust and C) was uneven.

Blockchain Technology✓ reviewed

Tech · Cryptography  ·  ~4 min read

Core Concept

INFO: Definition
A blockchain is an append-only distributed ledger composed of cryptographically linked blocks, where each block commits to its predecessor via a hash pointer, making retroactive tampering computationally infeasible without network-wide consensus.

Formally, treat each block $B_i$ as a tuple:

$$B_i = (h_{i-1},\ t_i,\ \text{root}(\mathcal{T}_i),\ \text{nonce}_i)$$

where $h_{i-1} = H(B_{i-1})$ is the parent's cryptographic hash, $t_i$ is a timestamp, and $\text{root}(\mathcal{T}_i)$ is the Merkle root of the transaction tree $\mathcal{T}_i$. Altering any $B_j$ invalidates $h_j$, which cascades to invalidate $B_{j+1}, \ldots, B_n$—requiring recomputation of all subsequent proof-of-work (or equivalent) plus capturing network consensus. This is the chain's tamper-evidence guarantee.

The Merkle tree structure provides $O(\log n)$ inclusion proofs: a lightweight client can verify a single transaction without downloading the full block, crucial for scalability.

Why It Matters

Blockchain solved the double-spend problem for digital cash—previously requiring a trusted central authority—by replacing institutional trust with cryptoeconomic incentive alignment. This is architecturally significant: it instantiates a trustless system from untrusted peers.

Key application domains:

  • Cryptocurrency — Bitcoin, Ethereum: permissionless value transfer
  • Smart contracts — Turing-complete execution (Ethereum EVM) enabling DeFi, DAOs, NFTs
  • Supply chain provenance — Immutable audit trails without a central record-keeper
  • Digital identity & credentialing — Self-sovereign identity (SSI) systems
  • Payment rails — Settlement finality without correspondent banking
TIP: Broader systems connection
Blockchain is a specialization of Byzantine Fault Tolerant (BFT) replicated state machines. The novelty of Nakamoto consensus was achieving BFT in an open, permissionless setting using proof-of-work as a Sybil-resistance mechanism—something classical BFT protocols (PBFT, Tendermint) assume away by requiring known validator sets.

Key Details

  1. Consensus Mechanisms differ fundamentally in trust model. Proof-of-Work (PoW) ties voting power to physical compute; Proof-of-Stake (PoS) ties it to bonded capital with slashing conditions. PoS reduces energy cost by ~99.9% but introduces different attack surfaces (long-range attacks, nothing-at-stake).
  1. Forks are real and consequential. Blockchains are not immutable in an absolute sense—a sufficiently resourced adversary controlling >50% of hashrate (PoW) or stake (PoS) can reorg the chain. The security guarantee is probabilistic: after $k$ confirmations, reversion cost exceeds rational attacker budget.
  1. Merkle trees enable SPV clients. Simple Payment Verification uses $O(\log n)$ sibling hashes to prove inclusion—this is what makes light wallets possible and is a key scalability primitive.
  1. Permissioned vs. permissionless is an architectural spectrum. Permissioned chains (Hyperledger Fabric, Quorum) trade decentralization for throughput and privacy—meaningful for enterprise use cases, but they sacrifice the Sybil-resistance property that motivates the design in the first place.
  1. The trilemma frames all design tradeoffs. Vitalik Buterin's scalability trilemma posits you can optimize at most two of {decentralization, security, scalability}. Layer-2 solutions (rollups, state channels) attempt to escape it by moving execution off-chain while inheriting L1 security.
WARNING
Note: this information may be dated — the Layer-2 ecosystem (ZK-rollups, optimistic rollups) and PoS adoption (Ethereum Merge, 2022) are rapidly evolving; specific protocol details may have shifted.

Current State

Active research fronts as of 2026:

  • ZK-SNARKs / ZK-STARKs for rollups — validity proofs compress batch transactions; STARK-based systems (StarkNet, Polygon zkEVM) avoid trusted setups at cost of larger proof size
  • Cross-chain interoperability — bridges remain a primary attack surface; formal verification of bridge contracts is an open problem
  • MEV (Maximal Extractable Value) — miner/validator ordering power creates front-running and sandwich attacks; PBS (Proposer-Builder Separation) and encrypted mempools are proposed mitigations
  • Formal verification of consensus — mechanized proofs of safety/liveness for PoS protocols under asynchronous network models

  1. The permissioned paradox: If a permissioned blockchain requires trusting the entity that controls membership, what security properties does it actually provide over a traditional replicated database like CockroachDB or Spanner—and when, if ever, is it the right tool?

Formal Verification✓ reviewed

Tech · Software Engineering  ·  ~5 min read

Core Concept

INFO
Formal Verification is the act of proving or disproving the correctness of a hardware or software system with respect to a formal specification, using mathematical proof techniques. The key distinction from testing: verification is exhaustive over all possible inputs/states; testing is finite sampling.

The core setup is a triple:

$$ \vdash \{P\}\; S\; \{Q\} $$

(a Hoare triple) asserting that if precondition $P$ holds before executing $S$, then postcondition $Q$ holds after — a formulation that generalises across most verification paradigms. A full system verification amounts to constructing a formal proof that a mathematical model of the system satisfies a formal property, then arguing the model faithfully represents the implementation.

Mathematical objects used to model systems span a wide spectrum:

  • Finite-state machines / labelled transition systems — for reactive/concurrent protocol analysis
  • Timed & hybrid automata — for real-time and cyber-physical systems
  • Process algebras (CSP, CCS, π-calculus) — for compositional concurrent reasoning
  • Operational / denotational / axiomatic semantics — for programming language properties
  • Petri nets / vector addition systems — for resource and concurrency modeling
  • Horn clauses — foundational to logic programming and CHC-based verification

Why It Matters

Formal verification occupies the top tier of assurance: EAL7 under Common Criteria, the highest evaluation assurance level in computer security certification. This makes it mandatory or strongly preferred in:

  • Safety-critical systems — avionics (DO-178C), automotive (ISO 26262), medical devices
  • Cryptographic protocols — TLS, SSH, and hardware crypto modules where a single bug is catastrophic
  • Hardware design — EDA flows routinely use model checking for RTL equivalence; Intel, AMD, and ARM all use formal tools post-Pentium FDIV bug (1994)
  • Compilers — a bug in a compiler invalidates every program it produces; hence CompCert, a formally verified C compiler with a machine-checked proof that compilation preserves semantics
TIP
The CompCert and seL4 projects demonstrate a crucial shift: formal verification is no longer purely academic. seL4's verified microkernel provides end-to-end proofs in Isabelle/HOL, covering functional correctness, integrity, and confidentiality — turning the kernel into a provably secure isolation layer.

Key Details

  1. Model Checking vs. Theorem Proving — The two dominant paradigms differ fundamentally in automation vs. expressivity. Model checking (SPIN, NuSMV, TLA+) exhaustively explores a finite state space algorithmically — push-button but limited by the state explosion problem. Theorem proving (Coq, Isabelle/HOL, Lean) handles unbounded systems but requires significant human guidance to construct proofs.
  1. State Explosion — The central adversary of model checking. For $n$ boolean variables, the state space is $2^n$. Binary Decision Diagrams (BDDs) and SAT/SMT solvers are the primary weapons for managing this; symbolic model checking replaces explicit state enumeration with BDD-encoded set operations.
  1. Specification Languages — Properties are expressed in temporal logics. LTL (Linear Temporal Logic) reasons over paths: $\square\, (p \Rightarrow \Diamond\, q)$ ("if $p$ ever holds, $q$ eventually follows"). CTL (Computation Tree Logic) quantifies over branching futures. CTL\* subsumes both but at greater computational cost.
  1. Abstraction and Refinement — Verifying a concrete implementation directly is often intractable. The standard technique is abstraction: verify an over-approximation, then if a counterexample is found, check whether it's spurious and refine the abstraction (CEGAR — CounterExample-Guided Abstraction Refinement).
  1. Trusted Computing Base — A proof is only as trustworthy as the proof checker. Proof assistants like Coq and Lean use a tiny, independently auditable kernel that checks every proof step, minimising the TCB. The de Bruijn criterion codifies this: the checker should be small enough to be verified by inspection.

  • LLM-assisted proof synthesis — using large language models to suggest Coq/Lean lemmas and tactics, dramatically reducing the human labour in proof construction
  • Verified AI/ML systems — formalising neural network properties (robustness, fairness) via abstract interpretation and SMT; tools like Marabou and α,β-CROWN target ReLU networks
  • Rust verification — leveraging Rust's ownership type system as a proof-friendly foundation; projects like Verus and Kani bring formal tools to systems programming without the overhead of proof assistants
  • Continuous verification in CI — integrating lightweight formal tools (property-based testing, bounded model checking with CBMC) directly into software pipelines
  • Verified distributed systems — IronFleet and similar projects extend mechanised proofs to distributed protocols like Paxos, where concurrency and partial failure compound complexity enormously

Discussion Prompts

  1. Where does the boundary lie? Model checking is decidable for finite systems but undecidable in general (cf. Rice's theorem); theorem proving is semi-decidable. Does this fundamental limit mean full-stack verification of complex systems is always a human-in-the-loop enterprise, or can proof synthesis close the gap?
  1. Specification is the hard problem. A system can be formally correct with respect to its specification and still fail catastrophically if the specification is wrong or incomplete. How should the verification community formalise the meta-problem of specification correctness, and what role do domain ontologies or contracts play?
  1. Economic viability at scale. CompCert and seL4 required person-decades of expert effort for relatively small codebases. Given that scale, how does the field justify the cost for non-safety-critical software — and does the rise of LLM-assisted proof assistants fundamentally change the cost calculus?
