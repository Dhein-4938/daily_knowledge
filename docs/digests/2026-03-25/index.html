Graph Neural Networks✓ reviewed

Tech · Machine Learning  ·  ~4 min read

Core Concept

INFO
A Graph Neural Network (GNN) is a neural architecture that operates directly on graph-structured data $G = (V, E)$, learning node, edge, or graph-level representations through iterative message passing between neighboring nodes.

At each layer $\ell$, a node $v$ aggregates messages from its neighborhood $\mathcal{N}(v)$ and updates its hidden state:

$$h_v^{(\ell+1)} = \text{UPDATE}^{(\ell)}\!\left(h_v^{(\ell)},\; \text{AGG}^{(\ell)}\!\left(\{h_u^{(\ell)} : u \in \mathcal{N}(v)\}\right)\right)$$

The choice of $\text{AGG}$ (sum, mean, max, attention-weighted) and $\text{UPDATE}$ (MLP, GRU cell, etc.) distinguishes the major architectures:

| Architecture | AGG scheme | Key property |

|---|---|---|

| GCN (Kipf & Welling) | Normalized sum | Spectral grounding |

| GraphSAGE | Sample + mean/LSTM | Inductive (unseen nodes) |

| GAT | Attention-weighted | Learnable neighbor weights |

| MPNN | Edge-featured sum | Chemistry-friendly |

| GIN | Sum + MLP | Maximally expressive (WL-equivalent) |

After $k$ layers, $h_v^{(k)}$ encodes the $k$-hop neighborhood structure around $v$.

Why It Matters

Graphs are the natural representation for a vast class of real-world systems — molecules, social networks, citation graphs, knowledge bases, physical simulations, and road networks — where Euclidean structure is absent and variable-size inputs are the norm. GNNs are the canonical tool for inductive reasoning over relational data.

TIP
GNNs unify seemingly disparate architectures. A standard 2D CNN is a GNN on a grid graph (nodes = pixels, edges = adjacency). A Transformer is a GNN on a complete graph with attention as the message-passing kernel. This perspective — part of geometric deep learning — reveals a single blueprint: exploit the symmetries of the input domain.

Applications with production impact:

  • Drug discovery: molecular property prediction, de novo generation (AlphaFold-adjacent pipelines)
  • Recommender systems: Pinterest, Uber Eats use GNN-based collaborative filtering
  • Combinatorial optimization: TSP, MaxCut approximation via learned heuristics
  • Physics simulations: learned simulators (e.g., GNS) replace numerical solvers for fluids and rigid bodies

Key Details

  1. Expressiveness ceiling — the WL hierarchy. Standard message-passing GNNs are at most as expressive as the 1-Weisfeiler-Lehman (1-WL) graph isomorphism test. They cannot distinguish certain non-isomorphic graphs (e.g., regular graphs of the same degree). Higher-order WL tests inspire $k$-GNNs, but at exponential cost.
  1. Over-smoothing and over-squashing. Stacking many layers causes node features to converge (over-smoothing). Independently, exponentially many nodes must route information through graph bottlenecks, causing information loss (over-squashing). These are distinct failure modes requiring different mitigations (residual connections, graph rewiring).
  1. Positional encodings on graphs. Unlike sequences or grids, graphs have no canonical node ordering. Learned or spectral positional encodings (Laplacian eigenvectors, random-walk statistics) break symmetry and boost expressiveness without full $k$-WL overhead.
  1. Edge and global features. The MPNN framework naturally handles edge attributes $e_{uv}$ by incorporating them into the message function $\phi(h_u, h_v, e_{uv})$. Graph-level tasks use a final $\text{READOUT}$ pooling step (sum/attention over all nodes).
  1. Heterogeneous graphs. Real knowledge graphs have typed nodes and edges. Relational GCN (R-GCN) and HAN use type-specific weight matrices, preventing spurious cross-type message mixing.

Current State

WARNING
Note: this information may be dated

Active research fronts as of early 2026:

  • Scalability: Full-batch message passing is $O(|E|)$ per layer — intractable for billion-edge graphs. Cluster-GCN, GraphSAINT, and neighbor sampling (GraphSAGE) are standard mitigations; scaling laws for GNNs remain underexplored.
  • Going beyond message passing: It is an open question whether expressive-enough GNNs must be built on message passing, or whether fundamentally different paradigms (e.g., subgraph GNNs, spectral methods) offer asymptotic advantages.
  • Graph Transformers: Architectures like GraphGPS and Exphormer hybridize attention with local message passing, effectively sidestepping the over-squashing bottleneck.
  • Generative GNNs: Diffusion models on graphs (e.g., DiGress) enable molecular generation with discrete graph structure — a rapidly maturing area.

WebAssembly✓ reviewed

Tech · Web Technologies  ·  ~4 min read

Core Concept

INFO
WebAssembly (Wasm) is a stack-based virtual instruction set architecture (ISA) defined by a portable binary format (.wasm) and an isomorphic text format (.wat). It specifies a structured execution model with formal semantics: a program is a module containing typed functions, linear memory, tables, and imports/exports — all validated before execution.

The type system is intentionally minimal: i32, i64, f32, f64 (with v128 for SIMD extensions). Instructions operate on a typed operand stack, and control flow is structured — no arbitrary jumps, only blocks, loops, and conditionals with explicit label targets. This design enables single-pass validation and makes formal proofs tractable.

The formal soundness theorem for the type system guarantees:

$$\text{If } \vdash m : \text{module} \text{ and } m \to^* c, \text{ then } c \text{ is not stuck}$$

i.e., well-typed modules don't reach undefined states — a property that C/C++ natively lacks.

Why It Matters

Wasm decouples compilation target from execution environment. A Rust crate, Go library, or C++ codec can compile once to .wasm and run in any compliant host — browser, server, edge node, embedded device — with near-native performance and memory-safety guarantees the host can enforce.

Key impact zones:

  • Web performance: Heavy workloads (codecs, physics engines, ML inference) run at speeds unreachable with JavaScript.
  • Sandboxed plugins: Host embedders can expose a capability API; the module can't escape the sandbox without explicit imports. Used heavily in eBPF-adjacent edge computing.
  • Universal compute substrate: WASI (WebAssembly System Interface) extends Wasm into a POSIX-like capability-oriented OS interface, enabling server-side and CLI use cases.
  • Language portability: Any language with an LLVM backend gains a Wasm target essentially for free.
TIP
Wasm's structured control flow is not a limitation — it's a feature. It allows JIT compilers to skip a full CFG reconstruction pass that native ISA decompilation requires, giving embedders faster compile-to-native pipelines (e.g., V8's Liftoff baseline compiler).

Key Details

  1. Linear Memory Model: A Wasm module has a flat, byte-addressable memory region starting at address 0. The host controls growth (memory.grow). Crucially, the Wasm program cannot access host memory — isolation is structural, not policy-based.
  1. The Validation Algorithm is O(n): The type-checking pass is linear in module size due to the stack-polymorphic type system and the absence of backward type dependencies. This matters for JIT-heavy environments where startup latency is critical.
  1. Component Model: The nascent Wasm Component Model (in active development) layers a rich IDL (WIT — Wasm Interface Types) atop core Wasm, enabling typed, language-agnostic inter-module calls. This is the foundation for Wasm-based microservices and plugin ecosystems.
  1. WASI and Capability Security: Rather than ambient authority (like POSIX), WASI passes capabilities as module imports. A module that isn't given a file-descriptor handle literally cannot open files — the sandbox is enforced at the ABI level, not via runtime checks.
  1. Threads and Atomics: The threads proposal adds shared linear memory and i32.atomic.* instructions, enabling parallel Wasm across Web Workers. The memory model aligns with C++ Memory Model (SC-DRF): data-race-free programs get sequentially consistent behavior.
WARNING
Note: this information may be dated — the Component Model and WASI Preview 2 specs were still stabilizing as of early 2026; implementation support across runtimes (Wasmtime, WasmEdge, V8) may have shifted significantly.

Current State

Active development fronts as of early 2026:

  • GC Proposal: Adds heap-allocated, garbage-collected structs and arrays, enabling high-fidelity compilation of managed languages (Kotlin, Dart, OCaml) without bundling a full GC runtime.
  • WASI Preview 2 / Component Model: Defining a stable ABI for composable, cross-language Wasm components — essentially a new universal FFI.
  • WasmGPU / GPGPU: Experimental proposals for exposing GPU compute (tied to WebGPU) through Wasm modules.
  • Wasm in non-web runtimes: Wasmtime (Bytecode Alliance) and WasmEdge are seeing adoption in serverless platforms (Fastly Compute, Fermyon Spin) as a lighter-weight alternative to containers.

Note: this information may be dated — the ecosystem is evolving rapidly and feature stabilization timelines shift frequently.

Blockchain Technology✓ reviewed

Tech · Cryptography  ·  ~4 min read

Core Concept

INFO: Definition
A blockchain is an append-only distributed ledger composed of cryptographically linked records (blocks). Each block commits to its predecessor via a hash pointer, forming an authenticated chain: $$B_i = \langle H(B_{i-1}),\ t_i,\ \text{root}(\mathcal{T}_i) \rangle$$ where $H$ is a collision-resistant hash function, $t_i$ is a timestamp, and $\text{root}(\mathcal{T}_i)$ is the Merkle root of the transaction set in block $i$.

The structure is essentially a hash-linked list: mutating any block $B_k$ invalidates $H(B_k)$, which cascades to invalidate $B_{k+1}, \ldots, B_n$. Retroactive tampering therefore requires recomputing every subsequent block and convincing the network to accept the revised chain — a bar set deliberately high by consensus mechanisms.

Transaction data within a block is organized as a Merkle tree: leaf nodes are transaction hashes, interior nodes are hashes of their children, and the root is a 32-byte commitment to the entire set. This allows $O(\log n)$ inclusion proofs, enabling lightweight clients (SPV) to verify membership without downloading full blocks.

Why It Matters

Blockchain solved a fundamental problem in distributed systems: double-spending without a trusted third party. Prior to Bitcoin (2008), preventing a digital asset from being spent twice required a central authority. Nakamoto's design replaced the authority with a decentralized consensus protocol, bootstrapping trust from economic incentives and cryptographic proofs.

Significance extends well beyond currency:

  • Smart contracts (Ethereum, 2015): Turing-complete code executed deterministically on every validating node, enabling trustless programmable agreements.
  • Supply chain provenance: Immutable audit trails without a single custodian.
  • Byzantine fault tolerance: Blockchains operationalize a BFT system tolerating up to $f < n/3$ (BFT) or $f < n/2$ (PoW Nakamoto) malicious nodes.
  • Digital identity and credentials: Self-sovereign identity anchored to a public ledger.
TIP: Connection to Distributed Systems Theory
Blockchain consensus is a concrete realization of the Byzantine Generals Problem. PoW sidesteps the classical BFT impossibility result (FLP, 1985) by introducing probabilistic finality and economic cost-of-attack rather than deterministic agreement — a subtle but critical departure from academic BFT literature.

Key Details

  1. Consensus Mechanisms: Proof-of-Work (PoW) achieves agreement via computational expenditure; the longest valid chain wins. Proof-of-Stake (PoS) replaces computation with collateralized stake, slashing dishonest validators. The security model differs fundamentally: PoW attacks cost energy; PoS attacks cost locked capital.
  1. Merkle Trees and Efficiency: The Merkle root enables stateless verification. A client can prove transaction inclusion with a path of $\lceil \log_2 n \rceil$ hashes — critical for mobile and IoT nodes that can't store gigabytes of chain data.
  1. Forks: A chain fork occurs when two valid blocks extend the same parent simultaneously. In PoW, the network resolves forks via the longest-chain (Nakamoto) rule. Hard forks introduce backward-incompatible rule changes and can permanently split chains (see: Bitcoin/Bitcoin Cash, 2017). Finality is thus probabilistic in Nakamoto consensus, not absolute.
  1. Permissioned vs. Permissionless: Public chains (Bitcoin, Ethereum) allow open participation; permissioned chains (Hyperledger Fabric) restrict node membership. Permissioned chains can run classical BFT protocols (e.g., PBFT, HotStuff) achieving deterministic finality, but sacrifice censorship resistance.
  1. Storage Scalability: Full nodes store every block since genesis. Bitcoin's UTXO model and Ethereum's world-state trie both grow monotonically. Stateless clients and Verkle trees are active research directions to decouple verification from full-state storage.

Current State

WARNING
Note: this information may be dated

Active research concentrates on scalability trilemma trade-offs (decentralization / security / throughput): Layer-2 rollups (optimistic and ZK-based) bundle transactions off-chain and post compressed proofs on-chain. ZK-rollups use succinct proofs (SNARKs/STARKs) to verify batches of thousands of transactions with a single on-chain proof, potentially achieving near-mainchain security at a fraction of the cost.

Ethereum's post-Merge PoS transition (2022) demonstrated large-scale migration of a live network from PoW to PoS, reducing energy consumption by ~99.95%. Danksharding (proto-danksharding via EIP-4844) introduces blob-carrying transactions to reduce rollup data costs, a stepping stone to full data availability sampling.

Formal Verification✓ reviewed

Tech · Software Engineering  ·  ~4 min read

Core Concept

INFO
Formal verification is the act of proving or disproving the correctness of a system with respect to a formal specification, using rigorous mathematical methods. It is distinct from testing: verification exhausts the entire state space (symbolically or deductively) rather than sampling it.

The central object is a proof obligation: given a model $M$ of a system and a property $\phi$ (expressed in some logic), show:

$$M \models \phi$$

where $\models$ denotes the satisfaction relation. The model $M$ might be a finite-state machine, a labelled transition system, or a denotational semantics of source code; $\phi$ might be a temporal logic formula, a Hoare triple, or a type invariant.

Why It Matters

Formal verification is the only technique that can provide unconditional guarantees (modulo the correctness of the specification and tool chain). This matters acutely in:

  • Safety-critical hardware: combinational and sequential circuits verified exhaustively before tape-out.
  • Cryptographic protocols: proving absence of authentication bypasses, replay attacks, etc.
  • OS kernels: seL4 is fully functionally correct and proved free of undefined C behavior.
  • Compilers: CompCert guarantees that compiled code preserves source semantics — no miscompilation.
  • Security certification: EAL7 (the highest Common Criteria assurance level) requires formal verification.
TIP
The verified compiler angle is underappreciated: most security arguments about code assume the compiler is trustworthy. CompCert collapses that assumption entirely, shifting the trusted computing base down to the proof assistant kernel (e.g., Coq's ~20 kLOC kernel).

Key Details

  1. Model vs. Reality gap. A proof is only as strong as the fidelity of the model. Abstracting floating-point as real arithmetic, or ignoring interrupt timing, can yield a "verified" system with real bugs. This is called the modeling gap and is often where failures actually occur.
  1. Decidability trade-offs. Full first-order arithmetic is undecidable (Gödel). Practical verification restricts to decidable fragments: propositional logic (SAT), linear arithmetic (SMT with LIA/LRA), or finite-state reachability (model checking). The expressiveness–decidability frontier shapes every tool's design.
  1. Deductive vs. Algorithmic approaches.
  • Model checking: exhaustive state-space exploration; automatic but suffers state explosion. Mitigated by BDDs, CEGAR, and bounded model checking (BMC) via SAT.
  • Theorem proving (Coq, Isabelle, Lean): arbitrary expressiveness but requires human proof engineering; proofs can run to hundreds of thousands of lines.
  • Abstract interpretation: computes sound over-approximations automatically; scales but may produce false positives.
  1. Specification is the hard part. Writing a complete, correct formal specification is often harder than the implementation. Incomplete specs give false assurance; over-constraining specs rule out valid implementations. The discipline of refinement (stepwise strengthening from abstract to concrete) is a principled response.
  1. Temporal logics for reactive systems. Properties of non-terminating systems (protocols, OS schedulers) are expressed in LTL or CTL:

$$\text{LTL safety: } \square \neg \text{bad} \qquad \text{LTL liveness: } \square\Diamond \text{response}$$

CTL model checking runs in $O(|M| \cdot |\phi|)$; LTL model checking is PSPACE-complete.

Current State

WARNING
Note: this information may be dated

Active research fronts include:

  • Interactive theorem provers maturing: Lean 4's mathlib and its meta-programming infrastructure are attracting both mathematicians and systems verification engineers.
  • Neural + formal hybrids: using LLMs to suggest proof steps or lemmas (e.g., Copilot for Lean), dramatically reducing human proof burden.
  • Verified AI systems: proving properties of neural networks (robustness, fairness bounds) via abstract interpretation tools like AI² and ERAN.
  • Hardware at scale: formally verifying modern out-of-order processors (e.g., work on RISC-V cores) remains an open engineering challenge due to microarchitectural state complexity.
  • Continuous verification in CI: integrating lightweight SMT-based checkers (e.g., Dafny, F*) into standard software pipelines.

  1. Specification completeness: If the formal spec itself can contain errors, how do we reason about the residual risk after verification? Is there a meta-level of assurance, or does trust ultimately bottom out in informal judgment?
