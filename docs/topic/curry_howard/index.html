Curry-Howard Correspondence✓ reviewed

Logic · Foundations  ·  ~4 min read

Core Concept

INFO: Definition
The Curry-Howard Correspondence is a deep structural isomorphism between systems of formal logic and typed computational calculi: propositions are types, proofs are programs, and proof normalization is computation (β-reduction).

The correspondence maps logical constructs to type-theoretic ones with precise fidelity:

| Logic | Type Theory |

|---|---|

| Proposition $A$ | Type $A$ |

| Proof of $A$ | Term $t : A$ |

| $A \Rightarrow B$ | Function type $A \to B$ |

| $A \wedge B$ | Product type $A \times B$ |

| $A \vee B$ | Sum type $A + B$ |

| $\bot$ (false) | Empty type $\mathbf{0}$ |

| $\top$ (true) | Unit type $\mathbf{1}$ |

| $\forall x{:}A.\ P(x)$ | Dependent product $\Pi_{x:A}\ P(x)$ |

| $\exists x{:}A.\ P(x)$ | Dependent sum $\Sigma_{x:A}\ P(x)$ |

To inhabit a type is to prove the corresponding proposition. A program that typechecks is a proof; a proof that normalizes is a terminating computation. The logical cut-elimination rule and β-reduction are the same thing, viewed through different lenses.

The isomorphism is grounded in intuitionistic (constructive) logic — classical logic with the Law of Excluded Middle ($A \vee \neg A$) corresponds to type systems with call/cc (continuations), a non-trivial and somewhat uncomfortable extension.

Why It Matters

The correspondence isn't a curiosity — it collapses three previously separate disciplines:

  • Programming languages: type systems gain logical meaning; well-typed programs are correctness certificates.
  • Proof theory: proofs become executable objects; proof assistants (Coq, Agda, Lean) exploit this directly.
  • Mathematics foundations: mathematical theorems can be run; existence proofs yield algorithms.

The three-way Curry-Howard-Lambek correspondence extends the picture: categories provide a third column, with objects as types/propositions and morphisms as programs/proofs. This unification is the backbone of categorical logic and topos theory.

TIP: Practical Payoff
Every time a Haskell type signature encodes a contract, or Lean verifies a proof by constructing a term, the correspondence is doing real work. Dependent types (Idris, Agda) push this further: the value of a term can appear in the type of another, allowing full program specifications as types.

Key Details

1. Normalization = computation.

Proof normalization (cutting out detours in a proof) corresponds exactly to β-reduction. Strong normalization of a type theory implies all programs terminate — which is why total languages like Agda reject general recursion.

2. Classical logic requires control operators.

Adding $\neg\neg A \Rightarrow A$ (double-negation elimination) to the type theory requires delimited continuations or call/cc. Parigot's λμ-calculus and Griffin's observation (1990) made this precise. Classical proofs correspond to programs that can inspect and manipulate the call stack.

3. Linear logic yields linear types.

Girard's linear logic ($A \multimap B$, resources used exactly once) corresponds to linear type systems — where a value cannot be duplicated or discarded. This is the logical foundation of Rust's ownership/borrow checker.

4. The correspondence is syntax-sensitive.

It lives at the level of proof terms, not just provability. Two different proofs of the same theorem are genuinely different programs with potentially different computational behavior. This is why proof relevance vs. irrelevance matters in dependent type theory.

5. Homotopy Type Theory (HoTT) extends it further.

In HoTT, types are spaces, terms are points, proofs of equality are paths, and higher-dimensional structure (homotopies) corresponds to proof transformations. The Univalence Axiom states that equivalent types are equal — a statement with no direct classical analogue.

$$\text{Univalence:}\ (A \simeq B) \simeq (A =_{\mathcal{U}} B)$$

Current State

WARNING: Note: this information may be dated

Active research threads as of early 2026:

  • Cubical type theory (cubical Agda, Redtt): constructive models of HoTT that make Univalence computable, resolving the prior situation where the axiom blocked computation.
  • Quantitative type theory (QTT): tracking resource usage in types, merging linear and dependent typing — foundation of Idris 2.
  • Proof mining: extracting computational content (algorithms, bounds) from classical mathematical proofs via the correspondence, used actively in numerical analysis.
  • Propositions-as-sessions: extending the correspondence to session types for concurrent processes, where propositions in linear logic correspond to communication protocols.

Discussion Prompts

  1. Classical vs. constructive: If adding $\neg\neg$-elimination forces continuations, what does this say about the computational meaning of classical existence proofs — do they actually compute anything useful?
  1. Normalization and totality: Strong normalization prevents general recursion, making Agda/Coq not Turing-complete as programming languages. Is this an acceptable trade-off, and how do real proof assistants escape it (e.g., Coq's Fixpoint guardedness checker)?
  1. HoTT and physics: The analogy between types as spaces and the role of homotopy in modern physics (topological field theory, cobordism) feels more than superficial. Is there a meaningful sense in which physical dualities (e.g., AdS/CFT) have a proof-theoretic shadow?
