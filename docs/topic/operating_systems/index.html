Operating System Design✓ reviewed

Tech · Systems  ·  ~4 min read

Core Concept

An operating system is privileged system software that mediates between application code and physical hardware, providing resource abstractions, isolation guarantees, and common services. Its design space is governed by a fundamental tension: efficiency (minimizing indirection) versus safety (enforcing protection boundaries).

The OS executes in a privileged execution mode (kernel mode / ring 0), while user applications run in unprivileged mode. Transitions between these modes occur via system calls (syscalls) — synchronous software-triggered traps — or interrupts — asynchronous hardware signals.

INFO
A syscall is not a function call. It triggers a mode switch through a gate (e.g., SYSCALL/SYSRET on x86-64), saving registers, switching stacks, and jumping to a kernel handler. Round-trip cost is typically 100–300 ns on modern hardware, often dominating I/O latency in fine-grained workloads.

The core abstractions an OS must provide:

| Abstraction | Underlying Resource |

|---|---|

| Process / Thread | CPU |

| Virtual Memory | Physical RAM + swap |

| File / inode | Persistent storage |

| Socket / pipe | Network / IPC |

| Device file | Peripheral hardware |

Why It Matters

OS design choices propagate through the entire software stack. Scheduler policy determines tail latency for cloud services. Memory allocator design shapes GC pause times. Filesystem semantics define the failure modes of databases.

$$\text{Throughput} \approx \frac{N}{\text{Latency}} \quad \text{(Little's Law)}$$

where $N$ is concurrency. The OS controls both $N$ (thread scheduling, connection limits) and latency (syscall overhead, lock contention), making it the primary determinant of system-level performance.

Beyond performance: OS design is the foundation of security. The kernel's trusted computing base (TCB) must be minimal — every line of kernel code is a potential privilege-escalation vector. This motivates microkernel architectures (Mach, seL4) despite their IPC overhead.

TIP
The seL4 microkernel is formally verified correct in Isabelle/HOL — its implementation provably matches its specification. This is the gold standard for TCB minimization: ~10K lines of C, fully proved. Compare to Linux at ~30M lines.

Key Details

1. Scheduling Algorithms and Their Trade-offs

Classical schedulers (Round Robin, MLFQ) optimize for average throughput. Real-time schedulers (EDF, Rate-Monotonic) provide deadline guarantees. The Completely Fair Scheduler (CFS) in Linux uses a red-black tree keyed on virtual runtime to approximate fair CPU sharing in $O(\log n)$ per operation.

2. Virtual Memory and the TLB

The MMU walks a multi-level page table (4-level on x86-64) to translate virtual → physical addresses. The TLB caches recent translations. A TLB miss costs ~30–100 cycles; a page fault (OS intervention required) costs thousands. Huge pages (2 MB, 1 GB) reduce TLB pressure at the cost of internal fragmentation.

3. Monolithic vs. Microkernel vs. Exokernel

  • Monolithic (Linux, XNU): all subsystems in one address space — fast IPC, large TCB.
  • Microkernel (seL4, QNX): minimal kernel, services as user processes — isolated but IPC-heavy.
  • Exokernel (MIT): kernel only enforces protection; library OSes implement abstractions — maximum flexibility, minimal abstraction overhead.

4. Synchronization Primitives

Spinlocks waste CPU on contention; mutexes sleep but involve kernel transitions. The futex (fast userspace mutex) in Linux uses an atomic compare-and-swap in userspace and only drops into the kernel on contention — the common uncontended path costs ~2 ns.

5. Filesystem Consistency

Journaling (ext4, NTFS), copy-on-write (ZFS, Btrfs), and log-structured (F2FS, NILFS) are the three dominant strategies for crash consistency. COW provides snapshots and checksumming "for free" but amplifies write I/O; journaling has lower write amplification but no inherent data integrity checks.

Current State

WARNING
Note: this information may be dated

Active research fronts include:

  • Unikernels (MirageOS, Unikraft): compile only necessary OS components into a single-address-space image for VMs, eliminating syscall overhead entirely for cloud-native workloads.
  • io_uring (Linux 5.1+): an async I/O interface using shared-memory ring buffers, allowing thousands of I/O operations to be submitted/reaped with zero syscalls in the steady state.
  • eBPF: allows verified user-supplied bytecode to run in kernel context, enabling programmable networking, tracing, and security policy without kernel patches — effectively a safe, JIT-compiled extension mechanism.
  • RISC-V and capability hardware (CHERI): hardware-enforced pointer capabilities that could fundamentally change how kernels enforce memory isolation, potentially making type-safe OS components trivially enforceable.
  • AI/ML workload scheduling: modern datacenter OSes must co-schedule GPU kernels, handle NUMA topology across accelerators, and manage memory bandwidth — problems ill-addressed by classical scheduler theory.

Discussion Prompts

  1. The exokernel hypothesis argues that OS abstractions should be mechanisms, not policies — that library OSes can outperform monolithic kernels for specialized workloads. Where does this argument break down at scale, and does unikernel adoption vindicate it thirty years later?
  1. io_uring vs. kernel-bypass: DPDK and RDMA bypass the kernel entirely for networking. io_uring reduces syscall frequency to near-zero but stays in-kernel. What does the persistence of both approaches reveal about the actual bottleneck in high-throughput I/O — is it the syscall itself, or context switching, or something else?
  1. Formal verification (seL4) proves correctness of implementation against specification — but who verifies the specification captures the right security model? Where does proof end and trust begin in a verified OS stack?
