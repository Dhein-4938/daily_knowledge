Blockchain Technology✓ reviewed

Tech · Cryptography  ·  ~4 min read

Core Concept

INFO: Definition
A blockchain is an append-only distributed ledger composed of cryptographically linked blocks, where each block commits to its predecessor via a hash pointer, making retroactive tampering computationally infeasible without network-wide consensus.

Formally, treat each block $B_i$ as a tuple:

$$B_i = (h_{i-1},\ t_i,\ \text{root}(\mathcal{T}_i),\ \text{nonce}_i)$$

where $h_{i-1} = H(B_{i-1})$ is the parent's cryptographic hash, $t_i$ is a timestamp, and $\text{root}(\mathcal{T}_i)$ is the Merkle root of the transaction tree $\mathcal{T}_i$. Altering any $B_j$ invalidates $h_j$, which cascades to invalidate $B_{j+1}, \ldots, B_n$—requiring recomputation of all subsequent proof-of-work (or equivalent) plus capturing network consensus. This is the chain's tamper-evidence guarantee.

The Merkle tree structure provides $O(\log n)$ inclusion proofs: a lightweight client can verify a single transaction without downloading the full block, crucial for scalability.

Why It Matters

Blockchain solved the double-spend problem for digital cash—previously requiring a trusted central authority—by replacing institutional trust with cryptoeconomic incentive alignment. This is architecturally significant: it instantiates a trustless system from untrusted peers.

Key application domains:

  • Cryptocurrency — Bitcoin, Ethereum: permissionless value transfer
  • Smart contracts — Turing-complete execution (Ethereum EVM) enabling DeFi, DAOs, NFTs
  • Supply chain provenance — Immutable audit trails without a central record-keeper
  • Digital identity & credentialing — Self-sovereign identity (SSI) systems
  • Payment rails — Settlement finality without correspondent banking
TIP: Broader systems connection
Blockchain is a specialization of Byzantine Fault Tolerant (BFT) replicated state machines. The novelty of Nakamoto consensus was achieving BFT in an open, permissionless setting using proof-of-work as a Sybil-resistance mechanism—something classical BFT protocols (PBFT, Tendermint) assume away by requiring known validator sets.

Key Details

  1. Consensus Mechanisms differ fundamentally in trust model. Proof-of-Work (PoW) ties voting power to physical compute; Proof-of-Stake (PoS) ties it to bonded capital with slashing conditions. PoS reduces energy cost by ~99.9% but introduces different attack surfaces (long-range attacks, nothing-at-stake).
  1. Forks are real and consequential. Blockchains are not immutable in an absolute sense—a sufficiently resourced adversary controlling >50% of hashrate (PoW) or stake (PoS) can reorg the chain. The security guarantee is probabilistic: after $k$ confirmations, reversion cost exceeds rational attacker budget.
  1. Merkle trees enable SPV clients. Simple Payment Verification uses $O(\log n)$ sibling hashes to prove inclusion—this is what makes light wallets possible and is a key scalability primitive.
  1. Permissioned vs. permissionless is an architectural spectrum. Permissioned chains (Hyperledger Fabric, Quorum) trade decentralization for throughput and privacy—meaningful for enterprise use cases, but they sacrifice the Sybil-resistance property that motivates the design in the first place.
  1. The trilemma frames all design tradeoffs. Vitalik Buterin's scalability trilemma posits you can optimize at most two of {decentralization, security, scalability}. Layer-2 solutions (rollups, state channels) attempt to escape it by moving execution off-chain while inheriting L1 security.

Current State

Active research fronts as of 2026:

  • ZK-SNARKs / ZK-STARKs for rollups — validity proofs compress batch transactions; STARK-based systems (StarkNet, Polygon zkEVM) avoid trusted setups at cost of larger proof size
  • Cross-chain interoperability — bridges remain a primary attack surface; formal verification of bridge contracts is an open problem
  • MEV (Maximal Extractable Value) — miner/validator ordering power creates front-running and sandwich attacks; PBS (Proposer-Builder Separation) and encrypted mempools are proposed mitigations
  • Formal verification of consensus — mechanized proofs of safety/liveness for PoS protocols under asynchronous network models

Discussion Prompts

  1. Consensus vs. finality: PoW achieves probabilistic finality while BFT-style PoS achieves absolute finality after a supermajority vote—what are the latency and liveness tradeoffs, and when would you prefer one over the other in a system design?
  1. The permissioned paradox: If a permissioned blockchain requires trusting the entity that controls membership, what security properties does it actually provide over a traditional replicated database like CockroachDB or Spanner—and when, if ever, is it the right tool?
  1. Trilemma escape routes: Layer-2 rollups inherit L1 security by posting state roots on-chain, but validity depends on data availability. How does danksharding (EIP-4844 and successors) address the data availability bottleneck, and what residual trust assumptions remain?
