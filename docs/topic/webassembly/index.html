Core Concept

INFO
WebAssembly (Wasm) is a portable, binary instruction format for a stack-based virtual machine. It defines a compact binary encoding (.wasm) and a human-readable text format (.wat), along with a formal embedding API — the JavaScript API and Web API — for host-environment integration.

Wasm is not a traditional ISA; it targets a virtual Harvard-architecture machine with linear memory, a value-typed operand stack, and structured control flow (no arbitrary jumps — only blocks, loops, and branches). Programs execute in a sandboxed environment with no implicit access to the host.

The core type system is minimal:

$$\text{valtype} ::= \mathtt{i32} \mid \mathtt{i64} \mid \mathtt{f32} \mid \mathtt{f64}$$

Extended proposals add v128 (SIMD), funcref/externref (reference types), and garbage-collected structs (GC proposal). The module format bundles typed functions, a linear memory, tables, globals, and import/export declarations into a self-describing unit.

Why It Matters

Wasm closes the gap between native performance and safe, portable execution. Key use cases:

  • Browser compute: Video codecs, game engines, physics simulations, and cryptographic libraries run near-native speed without plugins.
  • Edge / serverless: Runtimes like Wasmtime, WASMer, and WAMR execute Wasm outside browsers with <1 ms cold-start times — a decisive advantage over container-based functions.
  • Language universality: C, C++, Rust, Go, Kotlin, and Python all compile to Wasm; the module format is language-agnostic by design.
  • Plugin architectures: Wasm's sandbox makes it an ideal untrusted plugin substrate (see Envoy filters, Extism, and VS Code's WASI-based extension host).
TIP
Wasm's structured control flow is not a limitation — it's what enables one-pass validation in $O(n)$ time and makes JIT compilation significantly simpler than for x86. The spec mandates that any compliant engine must validate a module before execution, making supply-chain exploits structurally harder.

Key Details

1. Linear Memory Model

Wasm memory is a contiguous, byte-addressable ArrayBuffer (initially 0–4 GiB in wasm32, extended to 64-bit in wasm64). Bounds checks are explicit in the bytecode; hardware memory protection via guard pages is a common JIT optimization. Pointer arithmetic is therefore auditable, unlike native binaries.

2. WASI — The System Interface

The WebAssembly System Interface (WASI) provides a capability-based POSIX-like API for file I/O, sockets, clocks, and randomness. Rather than ambient authority, WASI passes pre-opened file descriptors explicitly — a deliberate application of the principle of least privilege at the ABI layer. WASI Preview 2 (Component Model) restructures this around typed interfaces described in WIT (Wasm Interface Type).

3. The Component Model

The Component Model lifts Wasm from a low-level binary format to a composable module system. Components expose and consume typed interfaces (records, variants, streams) using the Canonical ABI, enabling language-agnostic linking without FFI boilerplate.

4. Compilation Tiers

Production engines (V8, SpiderMonkey, Wasmtime) use a two-tier strategy: a fast baseline compiler (Liftoff / Cranelift single-pass) for immediate execution, and an optimizing compiler (TurboFan / Cranelift full) running concurrently. This amortizes compilation latency while converging to near-native throughput.

5. Security Boundary

The sandbox enforces memory isolation, control-flow integrity (structured CFG), and type safety at the bytecode level. Spectre-class attacks remain a concern — engines mitigate via retpoline, serializing memory accesses, and disabling SharedArrayBuffer/Atomics under threat models.

WARNING
Note: this information may be dated — the Component Model and WASI Preview 2 ecosystem were still maturing as of early 2026; toolchain support (especially for languages beyond Rust and C) was uneven.

Current State

Active standardization tracks as of 2026:

| Proposal | Status | Impact |

|---|---|---|

| GC (Garbage Collection) | Shipped in V8/FF | Enables managed languages (Kotlin, Dart) without bundled GC |

| Threads + Atomics | Phase 4 | Shared ArrayBuffer, wait/notify — enables pthreads |

| Tail Calls | Phase 4 | Enables functional languages without stack overflow |

| Exception Handling | Phase 4 | Native try/catch in Wasm |

| Memory64 | Phase 4 | 64-bit linear address space |

| WASI Preview 2 | Active | Component Model, async I/O via wit-bindgen |

The Component Model is arguably the most architecturally significant shift — it repositions Wasm from "portable C ABI" to a universal composition layer analogous to what Docker did for deployment.

Discussion Prompts

  1. Composition vs. isolation: The Component Model enables fine-grained inter-component calls, but each component maintains its own linear memory. How does this compare to the zero-copy IPC approaches in microkernel OS design (e.g., seL4 capabilities)? Is crossing component boundaries fundamentally expensive?
  1. Wasm vs. eBPF: Both are sandboxed bytecode formats for untrusted code in trusted environments. eBPF lives in the kernel; Wasm in userspace runtimes. What does each design reveal about the trust model and performance constraints of its target environment?
  1. The GC proposal tension: Adding GC to Wasm lets managed-language runtimes avoid bundling their own collector — but it introduces a shared, host-managed heap that partially dissolves the clean linear-memory sandbox. Where does this leave the security and predictability story for latency-sensitive embedded Wasm targets?
