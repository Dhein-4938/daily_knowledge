Distributed Systems✓ reviewed

Tech · Systems  ·  ~4 min read

Core Concept

A distributed system is a collection of independent computing nodes that appear to users as a single coherent system. Nodes communicate exclusively via message passing — there is no shared memory, no shared clock, and no assumption of reliable delivery.

INFO: Formal Definition
A system is distributed if the failure of a machine you've never heard of can render your own program incorrect. — Leslie Lamport. More formally: a set of processes $P = \{p_1, p_2, \ldots, p_n\}$ connected by a network, where each $p_i$ has a local state $s_i$ and transitions occur via send/receive events.

The three irreducible challenges:

  1. Concurrency — events at different nodes interleave non-deterministically
  2. No global clock — wall-clock synchronization is bounded by network jitter (NTP ~ms, PTP ~μs); logical time fills the gap
  3. Partial failure — subsets of nodes can fail silently while others continue, making it impossible to distinguish a slow node from a dead one

Logical time addresses the clock problem. Lamport timestamps assign scalar $T$ such that $e \to f \Rightarrow T(e) < T(f)$. Vector clocks extend this to capture causality:

$$ VC(e)[i] = \max(VC(e)[i], VC(f)[i]) \;\forall i, \quad \text{on receive of } f $$

Vector clocks enable causal delivery and are foundational to CRDTs and distributed databases.

Why It Matters

Distributed systems are the substrate of nearly all modern computing infrastructure: cloud platforms, microservices, CDNs, blockchain, multiplayer games, and scientific computing grids. The central value proposition is horizontal scalability — adding nodes increases capacity at near-constant marginal cost, which is qualitatively different from vertical scaling's diminishing returns.

TIP: Scalability vs. Performance
Scalability is not the same as raw performance. A system is scalable where the marginal cost of additional workload is nearly constant (Brooker). A single fast machine may outperform a distributed one at low load — the crossover point is where distributed systems earn their complexity premium.

The field also disciplines thinking about fault models: crash-stop, crash-recovery, Byzantine. Most production systems assume crash-stop but design for crash-recovery; Byzantine tolerance (e.g., BFT consensus) is reserved for adversarial settings like blockchains.

Key Details

1. CAP Theorem and its nuances

A system cannot simultaneously guarantee Consistency, Availability, and Partition tolerance when a network partition occurs. In practice, partitions are rare but inevitable, so systems choose CP (e.g., HBase, Zookeeper) or AP (e.g., Cassandra, DynamoDB). PACELC refines this: even without partitions, there's a latency/consistency tradeoff.

2. Consensus is hard

The FLP impossibility result proves that in an asynchronous system with even one crash-faulty process, no deterministic algorithm can guarantee consensus. Real systems escape this via partial synchrony assumptions (Paxos, Raft) or randomization.

3. Fallacies of Distributed Computing

Peter Deutsch's eight fallacies — the network is reliable, latency is zero, bandwidth is infinite, topology doesn't change, etc. — enumerate the assumptions that cause distributed systems to fail in practice. Each is a design trap.

4. Replication strategies

Single-leader, multi-leader, and leaderless replication each make different tradeoffs. Leaderless (Dynamo-style) uses quorum reads/writes: $R + W > N$ ensures overlap, but anti-entropy and read-repair are required to reconcile divergent replicas.

$$R + W > N \implies \text{at least one node in quorum has latest write}$$

5. CRDTs enable coordination-free consistency

Conflict-free Replicated Data Types are data structures where all concurrent updates commute. State-based CRDTs require merge to be idempotent, commutative, and associative. They allow AP systems to offer useful guarantees without synchronization.

The CRDT landscape and production tooling (e.g., Automerge, Yjs) evolve rapidly; specific library recommendations may not reflect the current state of the art.

Current State

Active research areas include: geo-distributed consensus (minimizing cross-region latency in Paxos variants like EPaxos, WPaxos); serverless and disaggregated storage (separating compute from state); verifiable distributed systems (using TLA+ and Coq to formally verify protocols); and FoundationDB-style deterministic simulation testing for finding bugs that escape unit tests.

Discussion Prompts

  1. The FLP result assumes a fully asynchronous model — how do real systems (Raft, Zookeeper) escape it, and what assumptions do they actually require?
  2. CRDTs trade expressiveness for coordination freedom. What classes of application semantics are fundamentally incompatible with CRDT-style eventual consistency?
  3. Serverless platforms abstract away node topology — does this actually simplify distributed reasoning for the developer, or does it merely obscure the same failure modes?
