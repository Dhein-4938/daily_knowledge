Distributed Systems✓ reviewed

Tech · Systems  ·  ~4 min read

Core Concept

A distributed system is a collection of autonomous computational components, located on distinct networked machines, that coordinate exclusively via message passing to present a unified service or solve a common problem. The key distinction from parallel computing is _autonomy under partial failure_: components may fail independently without bringing down the whole.

INFO: Formal Characterization
A distributed system satisfies three defining properties: 1. No shared memory — state is exchanged only through messages 2. No global clock — components have only local time; ordering requires protocols 3. Independent failure — any subset of nodes may crash or be partitioned at any time

The three canonical challenges follow directly from this definition:

$$\text{Consistency} + \text{Availability} + \text{Partition tolerance} \Rightarrow \text{pick 2 (CAP theorem)}$$

The CAP theorem (Brewer, 2000; formally proved by Gilbert & Lynch, 2002) is the central impossibility result: under a network partition, a system must choose between returning a consistent answer or returning any answer.

Why It Matters

Distributed systems are the foundational substrate of essentially all modern infrastructure: microservices, cloud platforms, CDNs, consensus ledgers, large-scale ML training, and massively multiplayer games all instantiate this model. The incentive is threefold:

  • Horizontal scalability — marginal cost of additional workload approaches constant as capacity is added in commodity units
  • Fault tolerance — replication and redundancy eliminate single points of failure
  • Geographic locality — data can be placed close to consumers
TIP: Scalability ≠ Performance
Marc Brooker's definition is precise: scalability is about marginal cost, not raw throughput. A system that doubles cost for each doubling of load is not scalable even if it's fast. Serverless architectures approximate ideal scalability but total cost of ownership (cold starts, egress, orchestration overhead) must be weighed separately.

Key Details

1. The Lack of a Global Clock

Physical clocks drift; NTP provides only loose synchronization. Lamport clocks ($C(a) < C(b)$ if $a \to b$) capture happens-before ordering. Vector clocks extend this to full causal tracking across $n$ nodes: each node $i$ maintains $V_i[1..n]$, incrementing $V_i[i]$ on send and merging on receive.

2. Consensus Is Hard

The FLP impossibility result (Fischer, Lynch, Paterson, 1985) proves that no deterministic protocol can guarantee consensus in an asynchronous system with even one crash failure. Practical systems work around this via timeouts (Paxos, Raft) or probabilistic termination.

3. Fallacies of Distributed Computing

Peter Deutsch's eight fallacies (the network is reliable; latency is zero; bandwidth is infinite; the network is secure; topology doesn't change; there is one administrator; transport cost is zero; the network is homogeneous) remain the most productive checklist for auditing system designs.

4. Consistency Models Form a Lattice

From strongest to weakest: linearizability → sequential consistency → causal consistency → eventual consistency. Weaker models permit higher availability and lower latency but shift correctness burden onto application logic.

5. Message-Passing Mechanisms

Implementations span pure HTTP, RPC frameworks (gRPC, Thrift), and durable message queues (Kafka, RabbitMQ). Choice affects latency, ordering guarantees, backpressure semantics, and replay capability.

Current State

WARNING: Note: this information may be dated
Active research frontiers include: geo-distributed consensus with WAN-aware Paxos variants; disaggregated storage/compute architectures; and CRDTs (Conflict-free Replicated Data Types) as algebraic structures for coordination-free consistency. Serverless and "function mesh" paradigms are actively colliding the distributed-systems and PL communities.

Recent work on LLM-based agent frameworks (e.g., AgentFactory, 2026) applies distributed-systems thinking to autonomous agents: subagents as independent, failure-isolated units that communicate via standardized interfaces and accumulate executable knowledge — a direct instantiation of the distributed programming model in the AI layer.

Discussion Prompts

  1. CAP forces a binary choice, but PACELC extends this to latency tradeoffs even when no partition exists — does PACELC make CAP obsolete as a design tool, or do they address different questions?
  1. The FLP impossibility assumes a purely asynchronous model; Raft and Paxos rely on timeouts (effectively introducing partial synchrony). What does this mean for the correctness guarantees we actually get in production?
  1. As LLM-based multi-agent systems scale, they exhibit the same three challenges (concurrency, no global clock, independent failure). Which distributed-systems primitives map cleanly onto agent coordination, and which require new abstractions?
