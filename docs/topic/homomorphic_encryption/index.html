Homomorphic Encryption✓ reviewed

Tech · Cryptography  ·  ~3 min read

Core Concept

INFO: Definition
Homomorphic encryption (HE) is a cryptosystem that permits arbitrary computations on ciphertexts such that the result, when decrypted, matches the output of the same computation on the plaintext. Formally, for encryption $\mathcal{E}$ and operation $\circ$: $$\mathcal{D}\!\left(\mathcal{E}(a) \otimes \mathcal{E}(b)\right) = a \circ b$$ where $\otimes$ is the "homomorphic" counterpart of $\circ$ in ciphertext space.

The scheme hierarchy runs from weakest to strongest:

  • Partially Homomorphic (PHE) — supports one operation (e.g., only addition or multiplication). RSA is multiplicatively homomorphic; Paillier additively.
  • Somewhat Homomorphic (SHE) — supports both operations, but up to bounded circuit depth.
  • Leveled FHE — both operations to pre-specified depth $L$.
  • Fully Homomorphic Encryption (FHE) — unbounded circuit depth; arbitrary polynomial-time computation. First constructed by Craig Gentry (2009) via lattice-based cryptography.

Gentry's breakthrough used ideal lattices and a bootstrapping procedure to reset accumulated noise, converting SHE into FHE.

Why It Matters

HE decouples data custody from computation: a cloud provider processes sensitive data without ever seeing it in the clear. This resolves a fundamental trust boundary that TLS and access control cannot touch — those protect data in transit and at rest, but plaintext must be exposed during computation.

Key application domains:

  • Privacy-preserving ML: run inference on encrypted patient records; model provider learns nothing about inputs.
  • Secure multi-party computation (MPC): HE is one mechanism to achieve MPC without interaction-heavy protocols.
  • Regulatory compliance: HIPAA, GDPR constraints softened when the processor is cryptographically blind.
  • Verifiable outsourcing: combine with SNARKs for verified computation on encrypted data.
TIP: The Privilege Escalation Angle
HE eliminates the attack surface where an adversary with OS-level or hypervisor-level access can intercept plaintext during processing. A compromised cloud host reads only ciphertext — an intrinsically different threat model from TEEs (Trusted Execution Environments), which rely on hardware trust.

Key Details

  1. Noise Growth & Bootstrapping — LWE-based schemes (BFV, BGV, CKKS) embed noise $e$ into ciphertexts. Each homomorphic multiplication roughly squares the noise magnitude. Bootstrapping homomorphically evaluates the decryption circuit to refresh noise — expensive ($\sim$seconds per bootstrap on modern hardware).
  1. CKKS for Approximate Arithmetic — The CKKS scheme sacrifices exact arithmetic for floating-point-friendly operations, ideal for ML inference. Errors are bounded but non-zero:

$$\|\mathcal{D}(\mathcal{E}(m_1)\otimes\mathcal{E}(m_2)) - m_1 \cdot m_2\| \leq \epsilon$$

  1. Security Foundations — Modern FHE rests on the hardness of Ring Learning With Errors (RLWE): given $(a, a\cdot s + e)$ with $a \in \mathbb{Z}_q[x]/f(x)$, recover secret $s$. This is believed quantum-resistant, unlike RSA/ECC.
  1. Parameter Selection Trade-offs — The polynomial degree $n$ and ciphertext modulus $q$ jointly govern security ($\lambda \approx n \log q / \text{bits}$) and performance. Larger $n$ → more noise headroom → deeper circuits, but $O(n \log n)$ NTT cost per operation.
  1. Ciphertext Expansion — A 32-bit plaintext may expand to kilobytes or megabytes of ciphertext. Bandwidth and storage overhead remain a practical obstacle for real deployments.

Current State

WARNING: Note: this information may be dated
FHE performance is improving rapidly. Libraries like OpenFHE, Microsoft SEAL, HElib, and Concrete (Zama) are active. Hardware accelerators (FPGAs, ASICs) targeting NTT and modular arithmetic are an open research frontier. The NIST PQC standardization process has standardized CRYSTALS-Kyber/Dilithium (lattice-based), creating ecosystem pressure that indirectly validates RLWE-based FHE security assumptions.

Active research threads include: reducing bootstrapping latency below 100ms, batching via SIMD slot packing (CRT-based), threshold FHE for distributed key management, and FHE-friendly neural network architectures (polynomial activations replacing ReLU).
