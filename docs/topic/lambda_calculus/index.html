Lambda Calculus✓ reviewed

Logic · Computation Theory  ·  ~4 min read

Core Concept

INFO: Definition
Lambda calculus (λ-calculus) is a formal system for expressing computation via function abstraction and application. Every program is either a variable, an abstraction, or an application — nothing else.

The full grammar fits in three lines:

$$ e \;::=\; x \;\mid\; \lambda x.\, e \;\mid\; e_1\; e_2 $$

where $x$ ranges over variables, $\lambda x.\, e$ binds $x$ in body $e$, and $e_1\; e_2$ applies $e_1$ to argument $e_2$.

Computation proceeds by β-reduction: substituting the argument for the bound variable.

$$ (\lambda x.\, e_1)\; e_2 \;\longrightarrow_\beta\; e_1[x \mapsto e_2] $$

A term with no further β-redexes is in β-normal form. An additional rule, η-reduction ($\lambda x.\, f\, x \longrightarrow_\eta f$ when $x \notin \text{FV}(f)$), captures extensional function equality.

Church introduced the untyped λ-calculus in the 1930s while investigating the foundations of mathematics; a logically consistent fragment was documented by 1940.

Why It Matters

Lambda calculus is Turing-complete — any Turing machine computation can be encoded in it, and vice versa. This makes it the minimal universal model of computation, smaller and more mathematically tractable than Turing machines.

Its influence is enormous:

  • Programming languages — Lisp, ML, Haskell, and most functional languages are essentially typed or untyped λ-calculi with syntactic sugar.
  • Type theory — the Curry-Howard correspondence maps λ-calculus terms to proofs and types to propositions, bridging logic and computation.
  • Denotational semantics — programs are modelled as mathematical functions, formalizing meaning independent of any machine.
  • Proof assistants — Coq, Agda, and Lean are all built on dependent type theories that extend λ-calculus.
TIP: Curry-Howard at a glance
A typed λ-term $t : A \to B$ is a proof that proposition $A$ implies $B$. Function application is modus ponens. This correspondence runs deep enough that proof checking and type checking are literally the same algorithm.

Key Details

1. Capture-avoiding substitution is non-trivial.

Naïve substitution breaks under variable shadowing. $(\lambda y.\, \lambda x.\, y)\; x$ must rename the bound $x$ before substituting, yielding $\lambda z.\, x$, not the broken $\lambda x.\, x$.

2. Fixed-point combinators enable recursion.

The untyped calculus has no primitive recursion, yet the Y combinator

$$Y \;=\; \lambda f.\,(\lambda x.\, f\,(x\,x))\,(\lambda x.\, f\,(x\,x))$$

satisfies $Y\, f \longrightarrow_\beta f\,(Y\, f)$, encoding arbitrary recursion. Notably, $Y$ cannot be typed in the simply-typed λ-calculus, which rules out non-termination by design.

3. Church encodings represent data as functions.

Booleans: $\mathbf{true} = \lambda t.\, \lambda f.\, t$, $\mathbf{false} = \lambda t.\, \lambda f.\, f$. Church numerals: $\overline{n} = \lambda f.\, \lambda x.\, f^n\, x$. Every data structure is ultimately a higher-order function.

4. Confluence (Church-Rosser theorem).

If $e \twoheadrightarrow_\beta e_1$ and $e \twoheadrightarrow_\beta e_2$, there exists $e_3$ with $e_1 \twoheadrightarrow_\beta e_3$ and $e_2 \twoheadrightarrow_\beta e_3$. Normal forms are unique when they exist, but not all terms terminate.

5. Reduction strategy matters.

Normal-order (leftmost-outermost) reduction finds a normal form whenever one exists; applicative-order (innermost-first, as in most languages) can diverge on terms that do have normal forms. Lazy evaluation is essentially call-by-need, a memoized normal-order strategy.

Current State

WARNING: Note: this information may be dated

Active research lives in the typed extensions:

  • Dependent types (Martin-Löf, CoC) unify types and terms, enabling proof-carrying code.
  • Linear and affine type systems (relevant to Rust Ownership Model) track resource usage at the type level, deriving from linear logic.
  • Gradual typing mixes static and dynamic guarantees in a single λ-calculus framework.
  • Optimal reduction (Lévy, Lamping) minimises β-steps by sharing redexes via interaction nets — still an area of implementation research.

Discussion Prompts

  1. Church encodings prove data is "just functions" — but is this a philosophical statement about the nature of data, or merely a formal trick? Where does the distinction break down?
  2. The Y combinator is untypable in simply-typed λ-calculus. Does the loss of self-application mean typed systems are less expressive, or do dependent and recursive types recover everything that matters?
  3. Normal-order reduction is theoretically complete but operationally expensive due to repeated substitution. How do modern compilers (GHC's STG machine, for instance) reconcile lazy evaluation with efficient memory layout?
