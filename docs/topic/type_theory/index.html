Type Theory✓ reviewed

Logic · Foundations  ·  ~4 min read

Core Concept

Type theory is the study of formal systems in which every expression is assigned a type — a classification that governs what the expression is and what operations are valid on it. This is not merely bookkeeping: the type discipline itself carries logical content.

INFO: Formal Definition
A type theory is a formal system comprising a set of types $\tau$, a set of terms $t$, and a typing judgment of the form: $$\Gamma \vdash t : \tau$$ meaning "in context $\Gamma$, term $t$ has type $\tau$." Rules for forming valid judgments are called typing rules.

The foundational dichotomy is between simple and dependent type theories:

  • Simple type theory (Church, 1940): types are fixed independently of terms. Function types written $\alpha \to \beta$.
  • Dependent type theory (Martin-Löf, 1972–84): types can depend on terms. The type $\text{Vec}(n)$ of vectors of length $n$ is a canonical example. This enables propositions to be encoded as types via the Curry–Howard correspondence: a proof of proposition $P$ is a term of type $P$.

The identity type $\text{Id}_A(a, b)$ — whose inhabitants are proofs that $a = b$ — is the engine behind Homotopy Type Theory, where paths between points in a space model propositional equality.

Why It Matters

Type theory occupies a rare intersection of logic, computation, and mathematics:

  1. Foundation of mathematics: Martin-Löf Type Theory (MLTT) and the Calculus of Inductive Constructions (CIC) are serious rivals to Set Theory (ZFC) as foundational frameworks — constructive by default, proof-relevant, and computation-aware.
  2. Proof assistants: Rocq (Coq) runs on CIC; Lean 4 is built on a dependent type theory called the Calculus of Constructions with universe polymorphism. These tools machine-check real mathematical proofs.
  3. Programming language design: Hindley-Milner type inference underlies ML and Haskell. Dependent types appear in Agda, Idris, and increasingly in industrial languages.
TIP: Propositions as Types
The Curry–Howard correspondence is not an analogy — it is an isomorphism. $\Pi$-types are universal quantifiers; $\Sigma$-types are existential quantifiers; the empty type $\bot$ is falsehood; the unit type $\top$ is truth. Writing a program is writing a proof.

Key Details

  1. Universes and Russell's Paradox: Naïve type theory admits $\text{Type} : \text{Type}$, which leads to Girard's Paradox (analogous to Russell's). The fix is a universe hierarchy $\mathcal{U}_0 : \mathcal{U}_1 : \mathcal{U}_2 : \cdots$, preventing a type from containing itself.
  1. Definitional vs. propositional equality: Two terms can be definitionally equal (interchangeable by computation rules, checked algorithmically) or merely propositionally equal (connected by an inhabitant of the identity type). The gap between these notions is where much subtlety lives.
  1. Univalence Axiom: In Homotopy Type Theory, Voevodsky's Univalence Axiom states $(A \simeq B) \simeq (A = B)$ — equivalence of types is their identity. This is not provable in MLTT; it must be added, and Cubical Type Theory makes it definitionally true.
  1. Termination and decidability: Typed lambda calculi with strong normalization (e.g., System F, CoC) guarantee all well-typed terms terminate — at the cost of not being Turing-complete as a programming language. The tension between expressiveness and decidability is fundamental.
  1. Linear and substructural types: Restricting how variables can be used (used exactly once in linear types) gives resource-aware type systems. Rust's ownership model is an industrial instance of affine typing.

Current State

WARNING: Note: this information may be dated

Active frontiers include:

  • Cubical type theories (CCHM, Cartesian Cubical TT): give computational interpretations of Univalence, making HoTT implementable.
  • Two-level type theories (2LTT): separating a "fibrant" layer (homotopy-aware) from a "strict" layer for meta-reasoning.
  • Multimodal and graded type theories: tracking resource usage, phase distinctions (compile-time vs. runtime), and security levels within a single framework.
  • Proof assistant interoperability: projects like Dedukti aim to translate proofs across systems built on different type theories.

Discussion Prompts

  1. Univalence makes type equivalence identical to type equality — does this dissolve the classical distinction between "isomorphism" and "equality," or merely relocate it?
  2. Dependent type theories are not Turing-complete (to ensure termination). Is this a fundamental limitation for verified software, or an acceptable trade-off? How do Coq/Lean escape it in practice?
  3. Both type theory and set theory can serve as foundations for mathematics, yet they have different ontological commitments (proofs as objects vs. extension-based sets). What mathematical practice would look like if dependent type theory became the default foundation taught first?
